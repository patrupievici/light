# ZVELT — Open-Source Reference Audit (Phase 1)

_Clean-room study of mature AGPL fitness apps (wger, Liftosaur, Endurain, FitTrackee) vs ZVELT, by an 8-theme multi-agent pass (2026-06-21)._

**License safety:** references are AGPL and were studied via public docs / READMEs / architecture / behavior + general knowledge ONLY. No source, schema, tests, assets, or UI text were copied or ported. Every recommendation is an ORIGINAL implementation described as a requirement; distinctive implementation details are to be designed independently. Concepts/architecture are not copyrightable; code expression is — we reuse the former, never the latter.

**Scope:** 8 themes, 63 recommendations — 13 P0, 27 P1, 23 P2.

## Per-theme detail


### exercise-catalogue-ranking-substitution
**References:** wger (exercise DB architecture: base/translation split, primary+secondary muscles, variations grouping, CC-BY-SA data license, REST API, 30-language community translations) — studied via wger.de/readthedocs/GitHub public docs. Liftosaur (target vs synergist muscle model, type/equipment taxonomy, muscle-based exercise substitution, custom-exercise "copy from built-in") — studied via liftosaur.com/exercises and public GitHub discussions #123/#173.

**ZVELT now:** ZVELT's catalogue is a flat single-table model: backend/prisma/schema.prisma `Exercise` has name, single nullable `primaryMuscle` (free-text string, no enum), `equipment`, `rankModel` (WEIGHTED|BW_REPS|TIME), `bwStrengthFraction`, `category`, `movementPattern` + `secondaryPatterns` JSON, `fatigueScore`, `goalTags` JSON, `contraindications` JSON, `beginnerSuitable`, plus custom-exercise flags. Notably there is NO secondaryMuscle column and NO localization/translation table (exercise names are English-only, no i18n hits in exercises.ts/schema). Catalogue listing/search: backend/src/routes/exercises.ts (substring filters on name/muscle/equipment/pattern, paginated, plus a server-side ExerciseDB RapidAPI proxy at /db/* and an image proxy). AI-name→row matching: backend/src/lib/exercise-resolver.ts (exact → normalized → Dice-bigram fuzzy at 0.72). Media/GIFs: backend/src/lib/exercise-media.ts joins a second Postgres DB by normalized name. Ranking: backend/src/services/ranking.service.ts (Epley e1RM capped at 12 reps, SR=e1RM/BW, name-substring-driven SR thresholds → LP 0-699 → 7 tiers Iron..Olympian; BW_REPS uses inferred bodyweight fraction). Muscle map: app/lib/widgets/muscle_map_widget.dart + app/lib/services/muscle_recovery_service.dart (NSCA-based per-muscle recovery hours, keyed off primaryMuscle only via a hand-maintained _muscleToSlug map, 7-day window, SharedPreferences cache). There is NO exercise-substitution feature anywhere — the closest is deterministic-workout.service.ts which fills movement-pattern slots ranked by fatigueScore (composeSession) and equipment-compatibility.ts mapping equipment tags.


**Strong ideas (clean-room):**
- Two-layer exercise model (language-neutral 'base' + localized presentation): wger separates the anatomical/equipment facts of an exercise (muscles, category, equipment, variation group) from per-language names/descriptions/aliases. ZVELT should adopt the same SEPARATION as a requirement — a stable canonical exercise identity that owns muscles/pattern/equipment, with display name/aliases/description as a translatable layer — so the same Squat can show localized names without forking ranking/recovery logic.
- First-class secondary muscles as structured data: both references model primary (target) AND secondary (synergist) muscles per exercise. ZVELT only stores a single free-text primaryMuscle, so the muscle map and recovery cannot represent that e.g. a Deadlift also loads hamstrings/forearms. Requirement: add structured primary + secondary muscle lists drawn from a controlled vocabulary so the muscle map can show partial activation and recovery can debit secondary muscles at a reduced weight.
- Controlled muscle vocabulary / enum instead of free-text: wger uses a fixed muscle list shared across exercises and translations. ZVELT's primaryMuscle is free text and the Flutter side already has to hand-maintain a fragile _muscleToSlug translation map (muscle_recovery_service.dart) where any seed typo silently drops a muscle from the map. Requirement: a single canonical muscle taxonomy used by DB, recovery service, and SVG map so the mapping is total and validated.
- Exercise variation/alternative grouping: wger groups 'similar' exercises (wide-grip / reverse / etc.) and Liftosaur lets a custom exercise be cloned from a built-in (copying type+muscles+image). Requirement: ZVELT needs an explicit notion of exercise families/variations so the app can offer 'other versions of this lift' and so a custom exercise can inherit muscles/pattern/rankModel from a parent.
- Muscle-similarity-driven substitution: Liftosaur substitutes an exercise with one hitting similar muscles when equipment is missing. ZVELT has movement-pattern slot filling but NO user-facing 'swap this exercise' that respects muscles. Requirement: a substitution ranker that, given a source exercise + the user's available equipment, returns alternatives scored by shared movement pattern + primary/secondary muscle overlap + comparable fatigue, with an explainability string (consistent with ZVELT's 'why' principle).
- Equipment as structured many-valued data with a normalized vocabulary: wger/Liftosaur treat equipment as a controlled set; ZVELT stores a single equipment string and equipment-compatibility.ts hardcodes a tag→value switch. A normalized equipment vocabulary + per-exercise allowed-equipment set would make substitution and equipment filtering far more robust (e.g. an exercise doable with barbell OR dumbbell).
- Data provenance / curation metadata: wger tracks license, source URL, author, and review status per exercise/image. ZVELT mixes its own seed rows with a live ExerciseDB RapidAPI proxy and a second GIF DB with no provenance field. Requirement: store source + license + last-reviewed per catalogue row so curation and legal attribution are auditable.
- Community/curation review workflow concept: wger exercises move through a submit→review→accept lifecycle. Even a lightweight 'status: draft|reviewed' flag on ZVELT exercises (especially custom ones promoted to global) would raise catalogue quality without a full wiki.

**Where ZVELT is already as-good/better:**
- Strength ranking is far richer than either reference: ZVELT has a full e1RM (Epley, reps capped 1-12, WORK-set only) → strength-ratio → LP → 7-tier system with seasonal LP and per-exercise persistence (ranking.service.ts). wger/Liftosaur are trackers; they do not provide a competitive percentile/tier ranking engine.
- BW_REPS rank model with per-exercise bodyweight-strength-fraction (push-up ~0.64, pull-up 1.0) lets bodyweight movements be ranked on the same SR scale — neither reference attempts a unified weighted+bodyweight strength score.
- Movement-pattern taxonomy (13 patterns) plus secondaryPatterns, fatigueScore, goalTags, beginnerSuitable and contraindications already encode programming intelligence that drives deterministic + AI workout generation — richer than wger/Liftosaur's category/type tags for the purpose of auto-programming.
- Muscle RECOVERY modeling (per-muscle NSCA recovery windows, ready/recovering/untrained states, offline cache) is a behavior neither reference ships as a core feature; ZVELT's muscle map is recovery-aware, not just an anatomical reference.
- Anti-cheat (set_edit_audits) + anomaly flagging on >20% SR jumps and bodyweight validation (BW_REQUIRED/BW_INVALID) give ZVELT integrity controls that open trackers don't need or have.
- AI-name→catalogue fuzzy resolver (exercise-resolver.ts) with normalization + Dice-bigram matching is a genuinely useful bridge from free-form AI output to canonical rows that the references don't require.

**Missing edge cases / test gaps:**
- No secondary-muscle data means muscle_recovery_service.dart never debits synergists — a heavy Deadlift marks only 'lower back' (or whatever single primaryMuscle is set) recovering, leaving hamstrings/forearms/glutes shown as fully Ready, which is physiologically wrong and untested.
- _muscleToSlug in muscle_recovery_service.dart is a hand-maintained partial map; any seed primaryMuscle value not in the map (e.g. a new muscle string, plural/singular mismatch, casing) is silently dropped (slug==null → continue) with no test asserting every catalogue muscle resolves to a slug.
- Ranking thresholds are selected by name.includes() substring matching (getSRThresholds / getSRThresholdsBw / inferBwStrengthFraction). A renamed or localized exercise, or an exercise whose name lacks the magic substring (e.g. 'Romanian Deadlift' vs a custom 'RDL'), silently falls into the generic bucket — no test guards the name→threshold mapping and localization would break it entirely.
- No localization: if exercise names are ever translated, BOTH the ranking substring logic and the GIF-by-name join (exercise-media.ts) and the AI resolver break, because all three key off the English name string.
- Custom exercises are forced isRanked:false and given movementPattern 'skill_stability' default (exercises.ts /custom) with no way to inherit a parent exercise's muscles/rankModel — so a user's custom 'Front Squat variation' is invisible to ranking, recovery accuracy, and programming.
- Substitution gap: when composeSession finds zero equipment-compatible exercises for a slot it just `continue`s (drops the slot) and bails if <3 chosen — there's no muscle-based fallback, so a user with limited equipment can get an empty/failed suggestion rather than a substituted alternative.
- ExerciseDB RapidAPI proxy data (/db/*) and the catalogue exercises are two disjoint datasets with different muscle/equipment vocabularies and no reconciliation — secondaryMuscles exist in gif-coverage.json/ExerciseDB but never land in ZVELT's Exercise rows.
- No provenance/license field on Exercise rows despite ingesting third-party ExerciseDB + external GIF DB content — an attribution/compliance gap and a test/data-quality blind spot.

**Recommendations:**

### localization-accessibility
**References:** wger (wger-project: Django backend, Flutter client, Weblate-driven community i18n, separate exercise-translation data model with English fallback) — public docs (readthedocs), README, GitHub issues #327/#450, and general knowledge of the project. Plus general OSS Flutter a11y/i18n patterns (gen-l10n/ARB, MaterialApp localization delegates, RTL/Directionality, MediaQuery textScaler).

**ZVELT now:** ZVELT has a partial, ad-hoc localization layer and a surprisingly strong but uneven accessibility base.

Localization:
- `app/lib/l10n/app_strings.dart` is a flat `AppStrings` class of EN `static const` literals (auth, onboarding, workouts, profile). It is NOT wired to Flutter's intl/gen-l10n — comment at top literally says "Replace with Localizations when adding language picker."
- Only 4 files actually import/use `AppStrings.`; the vast majority of UI text is hardcoded English string literals inline (e.g. analytics copy in `app/lib/screens/analytics/training_metric_detail_screen.dart`, `weekly_effort_screen.dart`).
- `app/lib/main.dart` (ZveltApp.build, ~line 341) configures `MaterialApp` with NO `localizationsDelegates`, NO `supportedLocales`, NO `locale`. The app is effectively locale-agnostic / English-only at the framework level.
- Backend speaks Romanian: `backend/src/routes/auth.ts` returns 'Acest email este deja folosit', 'Email sau parola incorecta', 'Date invalide'; `backend/src/middleware/auth.ts` 'Token invalid sau expirat'; `backend/src/routes/ai.ts` 'Mesaje invalide'. The client patches this over with `app/lib/l10n/auth_error_messages.dart` (`authErrorToEnglish`), a brittle substring matcher (RO/EN keywords → EN copy). This is a real RO/EN split across the boundary.
- A genuine Romanian string leaks into the UI: `app/lib/screens/profile/heatmap_screen.dart` ~line 700: 'Începe un antrenament GPS\npentru a-ți vedea traseul pe hartă.'
- Dynamic content (exercises) is single-language: prisma `Exercise.name` is a lone `String` (`backend/prisma/schema.prisma` ~line 293); seeds use English names ('Bench Press', 'Squat' in `backend/prisma/seed.ts`/`seed-demo-user.ts`). No translation/locale table, no per-language name/description.
- `intl` is a dependency and used for `DateFormat` (e.g. `journal_tab.dart`, `journal_entry_screen.dart`, `imported_history_screen.dart`) but always WITHOUT a locale argument, so month/weekday names are forced English regardless of device.
- Units: `UnitSystem { metric, imperial }` exists in `app/lib/models/onboarding_models.dart` and onboarding copy ('Metric (kg, cm)'/'Imperial (lb, in)' in app_strings), but weight rendering is hardcoded 'kg' (e.g. `app/lib/models/exercise_load_policy.dart` `${weightKg} kg`).

Accessibility (genuinely good in places):
- `main.dart` builder clamps `textScaler` to max 1.3 (protects dense layouts) and honors `disableAnimations || reduceMotion`.
- `reduceMotion` is a real persisted user preference (`app/lib/services/settings_store.dart`, surfaced in `app/lib/screens/settings/preference_settings_screens.dart`).
- ~41 files use `Semantics`/`semanticLabel`. `app/lib/widgets/muscle_map_widget.dart` builds a thoughtful screen-reader summary of the body map (`_semanticSummary`) and humanizes muscle slugs.
- 44x44 touch-target awareness exists (comment in `social_plus_screen.dart`).


**Strong ideas (clean-room):**
- Separate UI language from content language. wger treats the interface locale (buttons, labels) and the exercise/content locale as two independent axes. ZVELT should model a user's UI locale separately from which language exercise names/descriptions are shown in, because a user may want an English UI but localized exercise names (or vice versa).
- Language-neutral base exercise + per-language translation records. wger's exercise is a language-agnostic concept (muscles, equipment, category, movement) with a separate set of translation rows (name, description, optional aliases) keyed by language. ZVELT's `Exercise` currently mixes the neutral attributes with a single `name` string. The requirement: store the canonical concept once and attach 0..n localized name/description records, resolved at read time by requested locale.
- Graceful English fallback for missing content translations. wger serves the high-frequency exercises in many languages and falls back to English when a translation is absent, rather than showing blanks or a missing-key. Requirement: every content-localization read path must deterministically fall back (requested locale -> base/English -> raw key) and never surface an empty or untranslated-token string to the user.
- Community/moderated translation contribution pipeline (concept only). wger uses Weblate + a moderation queue so the catalog scales to 30 languages without the core team translating everything. For ZVELT this argues for: translatable content should live in data (not code), and there should be an ingestion/override path (admin or future contributor) that does not require an app release to add a language for exercises.
- Translatable strings as data, not compiled constants. The central lesson of a mature i18n setup is that all user-facing copy lives in locale resource files keyed by stable IDs, with the framework selecting the active set at runtime. ZVELT's `AppStrings` static-const approach blocks ever adding a second language. Requirement: adopt a real key->message catalog (ICU-style, supporting plurals/gender/placeholders) resolved through the framework's localization delegate.
- ICU plural/select and placeholder support. Mature catalogs use ICU MessageFormat so '1 set' vs '5 seturi' (and gendered/select forms) are handled per-locale instead of via English-only string concatenation. ZVELT currently concatenates like 'Set ${i}: ${reps} reps' which cannot pluralize correctly in other languages. Requirement: pluralizable counts go through ICU plural messages.
- Locale-aware number/date/unit formatting. A localized app formats dates, decimals, and units per locale (e.g. decimal comma, 24h vs 12h, weekday names). ZVELT uses DateFormat without a locale and hardcodes 'kg'. Requirement: all formatting flows through a locale-aware formatter and a unit-system-aware weight/height/distance formatter wired to the user's metric/imperial choice.
- RTL readiness as a structural property, not a retrofit. wger ships RTL languages (Arabic, Hebrew, Persian). Achieving this cheaply later requires building directional-agnostic layouts now: use start/end (EdgeInsetsDirectional, AlignmentDirectional) instead of left/right, avoid hardcoded TextDirection.ltr around real content, and let the framework drive directionality. Requirement: audit/forbid left/right padding and stray Directionality(ltr) wrappers except where genuinely direction-fixed (charts/math).
- Make backend locale-agnostic at the contract level. wger returns stable, machine-readable identifiers and lets the client localize. ZVELT's backend currently returns human Romanian prose that the client must regex-translate. Requirement: backend error responses carry a stable error CODE (the spec already mandates `{ error, message, request_id }`); the client maps code -> localized copy, eliminating the brittle `authErrorToEnglish` substring matcher.
- Accessibility audit checklist baked into tests. The strong-but-uneven a11y state (great muscle-map semantics, but 0 a11y tests) suggests value in encoding a11y invariants as automated checks: minimum tap target, presence of semantic labels on icon-only buttons, contrast of token pairs, and layout survival at max text scale.

**Where ZVELT is already as-good/better:**
- Text-scale safety net: ZVELT proactively clamps `textScaler` to 1.3 in `main.dart` to keep dense strength-tracking layouts from breaking at extreme device font sizes while still honoring moderate accessibility scaling — a deliberate, app-specific a11y decision many OSS apps lack.
- First-class reduce-motion preference that is both OS-honored (`disableAnimations`) and user-overridable and persisted (`settings_store.dart` / `preference_settings_screens.dart`).
- Domain-specific semantic summaries: `muscle_map_widget.dart` synthesizes a real screen-reader narrative of the body heatmap (ready/recovering muscles) from already-loaded state — richer than the generic alt-text typical OSS fitness UIs provide.
- Explainability copy that doubles as accessibility value text (e.g. e1RM/volume explanations in `training_metric_detail_screen.dart`) gives screen-reader users meaningful context, aligning with ZVELT's explainability principle.
- Single canonical unit storage (weight_kg/bodyweight_kg) with client-side conversion already decided in CLAUDE.md — a cleaner i18n/units foundation than apps that store mixed units.

**Missing edge cases / test gaps:**
- No automated tests for localization or accessibility: 0 of ~12 files in app/test cover Semantics, locale switching, textScaler max, or Directionality. Regressions (like the heatmap RO leak) ship silently.
- Missing-translation behavior is undefined for dynamic content because there is no translation layer at all — when a second language is added, exercise names/descriptions have no fallback contract.
- DateFormat is called without a locale everywhere (journal_tab.dart, journal_entry_screen.dart, imported_history_screen.dart), so dates won't localize even after UI localization lands; also no test pins expected formatting.
- Pluralization is impossible with current concatenation ('reps', 'set N') — no edge-case handling for languages with multiple plural forms (e.g. Romanian's few/many, Russian/Polish/Arabic plural categories).
- Unit-system edge case: imperial is selectable in onboarding but the display path hardcodes 'kg' (exercise_load_policy.dart), so an imperial user sees metric — a correctness gap, not just cosmetic.
- Backend Romanian prose reaches the user if `authErrorToEnglish` doesn't match (its `return message` fallback) — any unmapped RO error string leaks raw to an English UI.
- No RTL coverage: stray `TextDirection.ltr` wrappers (journal_tab.dart ~502, social_plus_screen.dart ~1477, z_performance_trend.dart) and likely left/right paddings mean RTL would break layout; no golden test at TextDirection.rtl.
- Icon-only controls / charts: with only 3 Tooltip usages and many icon buttons, some interactive elements likely lack semantic labels — untested, so unknown coverage.
- Contrast of token pairs (text2 #A9B0C0 on bg #0B0D10, accent on elevated) is asserted by principle but not verified against WCAG AA programmatically; theming with user accent color could drop contrast below threshold.
- Layout survival at textScaler 1.3 is assumed safe by the clamp but never asserted by a golden/widget test at max scale.

### nutrition-body-measurements
**References:** wger (wger-project/wger) — nutrition plans, ingredient database (Open Food Facts import), nutrition diary, body weight log, and user-defined custom measurement categories. Studied via public README, readthedocs, GitHub issues (#422 Open Food Facts integration, #875 custom measurements) and general knowledge of wger's well-known data model. No source was read or copied.

**ZVELT now:** ZVELT nutrition is local-first with server mirroring. Daily logging lives in app/lib/services/nutrition_service.dart: a per-day SharedPreferences blob (entries[], waterMl, weightKg) keyed `${userId}_nutrition_Y_M_D`, with a robust dirty-flag offline pattern (_isDayDirty/_markDayDirty/_pushDayToServer) so offline adds AND deletes survive sync-down (getDay lines 840-893, syncHistoryFromServer lines 705-750). Backend persists the day as an opaque JSON payload in NutritionLogDay (schema.prisma model at line 84; routes in backend/src/routes/nutrition.ts PUT/GET /day, GET /days max-62-day batch). Food data comes from USDA FoodData Central, not a local ingredient DB: app/lib/services/usda_fdc_client.dart (direct in debug, server proxy in release via /v1/nutrition/usda/foods/search and /usda/food/:fdcId, key kept server-side, lines 89-201 of nutrition.ts). Macro parsing handles kcal/kJ, nutrient-name variants, branded GTIN equivalence for barcode scan (lookupBrandedByGtin, usda_fdc_client.dart 157-221; barcode UI in nutrition_barcode_scan_screen.dart). Portion heuristics (egg/slice/serving) in nutrition_food_labels.dart. Goals are derived from the user profile (dailyCalories/Protein/Carbs/Fat/WaterMl on UserProfile, schema lines 166-170) via NutritionGoals.fromServerProfile. Weekly AI meal PLANNING is server-backed: NutritionPlanDay table (schema line 98) + DeepSeek generation with macro periodization for training vs rest days (nutrition.ts 204-336), editable meal-plan JSON (PATCH /plan/day, protein swaps). Body measurements (chest/waist/arms/thighs/bodyFatPct) are on-device only in app/lib/services/body_measurements_service.dart — one entry per day, upsert-by-day, latestWithDelta helper, sanity bounds 10–300cm / 2–60%bf, with an explicit comment that NO backend endpoint exists yet. Wearable-derived body metrics (strain/recovery/sleep) are computed transparently in body_metrics_service.dart. Bodyweight is also captured inside the nutrition day payload (weightKg) and on UserProfile.bodyweightKg.


**Strong ideas (clean-room):**
- Generic, user-extensible measurement categories: rather than ZVELT's fixed columns (chest/waist/arms/thighs/bodyFatPct), let a user define their own tracked metric as (name, unit, optional direction-of-good) and then log dated numeric entries against it. This is wger's central measurement abstraction and removes the need to ship a schema change every time someone wants neck/hip/calf/forearm/blood-pressure/resting-HR. Requirement: a 'measurement category' concept + a time series of dated values per category, with the existing latest+delta and chart logic generalised to any category.
- A normalized, server-side ingredient catalog with stable IDs and provenance — wger keeps each ingredient as a first-class row (name, brand, per-100g macros, optional serving sizes, barcode, source). ZVELT currently re-derives a FoodItem from a live USDA hit every time. Requirement: cache resolved foods server-side keyed by source+sourceId (e.g. usda_fdc_<id>, off_<gtin>) so repeat lookups are instant/offline-capable, logged entries keep referencing a stable food even if the upstream API is unavailable, and admins can correct bad macro rows.
- Open Food Facts as a second food source for BRANDED/packaged products and barcodes — wger's primary ingredient source. USDA Branded coverage is weak outside the US; OFF is global and barcode-first. Requirement: add a second resolver behind the same FoodItem abstraction, queried as a fallback (or primary for barcode), with source labeling preserved so users see where a number came from.
- Separate the logged amount from the food's per-100g basis using explicit serving units — wger stores meal items as ingredient + amount + unit (grams OR a named serving). ZVELT stores grams plus loose portion heuristics. Requirement: a structured serving model (gramsPerServing, unit label) attached to each food so '2 eggs' or '1 slice' is exact and round-trips, not re-guessed from the product name each time.
- Reusable meal templates / 'log this meal again' — wger nutrition plans contain named meals (Breakfast/Lunch...) made of ingredient items the user can re-apply. ZVELT has AI-generated PLANS but the manual DIARY has no concept of saving a frequently-eaten meal. Requirement: let users save a set of logged entries as a named meal and re-log it in one tap, drastically cutting repeat-logging friction (Persona B 'log rapid').
- A transparent BMI / TDEE / calorie-goal calculator that explains its inputs — wger exposes BMI and a daily-calorie calculator. ZVELT has goals but they're set/edited as raw numbers. Requirement (fits the Explainability principle): derive a SUGGESTED calorie+macro target from bodyweight/height/age/sex/activity/goal and show the 'why', letting the user accept or override — never silently overwrite their chosen goal.
- Body-weight as its own first-class dated time series, decoupled from the nutrition day — wger has a dedicated weight log. ZVELT smuggles weightKg inside the per-day nutrition payload, which couples bodyweight history to nutrition logging and makes it awkward to chart weight without loading nutrition days. Requirement: a standalone bodyweight series (or fold it into the generic measurement-category model above) so weight trend is queryable independently and powers the ranking SR computation cleanly.

**Where ZVELT is already as-good/better:**
- Offline-first correctness is materially stronger than a typical self-hosted web app: ZVELT's dirty-flag protocol (nutrition_service.dart 619-702, 840-893) guarantees offline adds AND deletes survive a sync-down race, replaying local-up instead of letting a stale server day erase them. wger is server-rendered/online-oriented and has no comparable offline-conflict guard.
- Honesty/explainability discipline: body_metrics_service.dart refuses to fabricate scores (null when no signal), and body_measurements_service.dart explicitly never invents numbers. ZVELT's macro parser also handles kJ→kcal conversion and nutrient-name variants defensively (usda_fdc_client.dart 286-338).
- Server-side AI weekly meal planning with goal-aware macro periodization (training vs rest day calorie adjustment, nutrition.ts 241-265) is a feature wger does not offer — wger plans are manually assembled.
- Security posture on the food API: USDA key never ships in the release APK; all release traffic goes through an authenticated server proxy with upstream error sanitization and rate-limit passthrough (nutrition.ts 89-201). Cleanly separates concerns.

**Missing edge cases / test gaps:**
- No automated tests for nutrition at all: app/test has no nutrition_service or body_measurements test (confirmed via glob). The dirty-flag offline/sync logic — the most bug-prone code — is untested. wger ships extensive test fixtures around nutrition/measurement.
- weightKg validation mismatch: backend PutDaySchema allows weightKg 20–400 (nutrition.ts line 21) but project spec/bodyweight bound is 30–250; the nutrition-day weight and UserProfile.bodyweightKg can diverge with no reconciliation, and SR/ranking reads bodyweightKg.
- Body measurements are device-local with NO backend, NO cross-device sync, and NO export — a phone wipe loses all history. The file itself flags this as a TODO. GDPR right-to-erasure/export also can't reach this data through the server.
- Unit handling: measurements are stored only in cm and bodyfat %, with no lb/inch input path — the spec says unit conversion happens client-side, but there is no imperial entry UI for body measurements, unlike weight.
- Per-day overwrite loses intraday granularity: body_measurements_service.log upserts by day (overwrites), and nutrition entries carry loggedAt but the day blob is replace-only — no append-with-timestamp audit, so editing history is lossy.
- Food macro provenance/correction: a wrong USDA row (e.g. mislabeled energy) flows straight into logged calories with no way to correct or pin a preferred source; logged entries embed a snapshot FoodItem, so a later upstream fix never reaches old logs (acceptable) but a bad number can't be fixed either.
- Branded/packaged & non-US coverage gap: barcode lookup is USDA-Branded-only (usda_fdc_client.dart 157-221); many EU/UK products (the user is in England) won't resolve, and there's no OFF fallback, so the UI may say 'not in database' for real products.

**Strong ideas (clean-room):**
- Layered, independent activity visibility (FitTrackee model): separate privacy toggles for (a) workout data + title/records, (b) analysis = charts/elevation/segments, (c) map/coordinates. ZVELT currently has no per-activity GPS privacy at all — a single 'who can see this route' is too coarse. Requirement: a user should be able to publish a run's stats and a finish badge while keeping the actual GPS trace/map private (anti-stalking, home-address leakage). This is a strong privacy-by-default fit for ZVELT's stated principles.
- Map/coordinate fuzzing for start/end: hide the first/last N meters of any shared route (and snap the displayed start to a coarse grid) so a public route never reveals a user's home. FitTrackee/Strava both treat this as table-stakes. ZVELT shares raw route_points today.
- Extreme-speed outlier exclusion for max-speed and pace metrics (FitTrackee default-on, user-toggle): a single GPS error can spike 'max speed' to absurd values. RouteTracker already drops teleports from DISTANCE but max/avg speed and the pace chart still consume raw pos.speed. Requirement: derive instantaneous speed from accepted, filtered points and clamp/winsorize outliers before computing max/avg/pace.
- Server-side derivation of splits + records from the stored route, not just client display. FitTrackee computes record types per sport (farthest distance, best pace, highest ascent, longest duration, max speed). ZVELT has a SplitData *widget* and SplitsTable but no code that actually builds splits from route_points, and no 'personal records' surface for outdoor activities. Requirement: backend computes per-km splits, elevation gain, moving vs elapsed time, and personal-best records from the canonical route on save.
- File import (GPX/TCX/FIT) for outdoor activities. Both references center on import; ZVELT has zero import path (grep confirms no gpx/tcx/fit handling anywhere). This unlocks migration from Strava/Garmin/Watch exports and is a major acquisition lever. Requirement: validated multi-format import that normalizes into ZVELT's route_points + distance/elevation/duration model.
- GPS stream normalization as a shared, server-trusted pipeline. Both apps treat the recorded track as the source of truth and recompute metrics server-side. ZVELT trusts client-sent distance_m/duration_s/calories on POST /v1/activities and POST /cardio/complete with no cross-check against route_points — an anti-cheat hole given ZVELT's competitive segments/XP. Requirement: server recomputes distance/elevation/duration from route_points and rejects/flag-mismatches client-claimed values.

- Moving-time vs elapsed-time distinction. FitTrackee/Strava separate the two; ZVELT only tracks elapsed (_elapsedSec) and has a stationary AUTO-STOP but no auto-PAUSE, so a stop-light run shows slow average pace. Requirement: compute moving time (exclude sub-threshold-speed gaps) and surface both, plus moving-pace splits.
- Segment matching robustness: FitTrackee/Strava match by traversing the whole segment polyline, not just start/end proximity. ### privacy-export-deletion
**References:** Endurain (self-hosted, per-activity visibility, standard-format export, data-ownership philosophy) + FitTrackee (async queued data-export archive job with ready-notification, account deletion that removes DB rows AND uploaded files, per-workout privacy levels, CLI tooling for export requests)

**ZVELT now:** ZVELT already has a solid, test-covered first pass at GDPR.
- Right to erasure: `backend/src/routes/gdpr.ts` `eraseUser()` deletes the full dependent object graph children-before-parents inside a single `prisma.$transaction` (atomic — no orphans on partial failure), then deletes the user row last. Covered by `backend/src/routes/gdpr.test.ts` (asserts single transaction, ordering invariants, 500 ERASURE_FAILED on throw). DELETE /v1/me/account returns 204.
- Data portability: `backend/src/routes/profile.ts` GET /v1/me/export-data returns a single JSON document (`formatVersion:1`, `exportedAt`, `data`) with Content-Disposition attachment + Cache-Control private/no-store. `backend/src/routes/profile.export.test.ts` proves credentials (authIdentities, refreshTokens, passwordHash) are never SELECTed nor serialized, and 404 USER_NOT_FOUND when account gone.
- Per-type health consent ledger: `backend/src/routes/health-consent.ts` GET/POST /v1/me/health-consents, per consentType (steps, heart_rate, sleep, weight, hrv, blood_oxygen, active_energy, distance, workout, or 'all'), with granted flag, consentVersion, source, grantedAt, revokedAt. Schema `HealthConsent` (schema.prisma ~901) has @@unique([userId, consentType]) and onDelete:Cascade.
- Post/profile privacy: `backend/src/lib/post-visibility.ts` pure `canViewerSeePostPure` (private/friends/public, unknown defaults to restrictive); enforced in posts routes with 404 (not 403) so forbidden posts are indistinguishable from missing ones (posts.privacy.test.ts proves byte-identical 404 + showActivityFeed short-circuit). Privacy-by-default: discoveryOptIn=false, feedFriendsOnly=true, privacyDefault='friends'.
- change-password requires current password and invalidates all refresh tokens.


**Strong ideas (clean-room):**
- Asynchronous export-as-a-job with a download artifact and ready-notification (FitTrackee pattern): instead of building the whole export inline in one request, accept an export REQUEST, enqueue it, build the archive in the background, store it, and notify the user (push/in-app) when it is ready to download. Removes the request-timeout / memory-blowup risk for accounts with thousands of sets and lets the export include binary media.
- Export should be a self-describing ARCHIVE (manifest + content + media), not just a JSON blob: include a manifest (schema version, generated-at, record counts per entity, checksum) plus the user's uploaded binaries (avatar, post photos, GPS tracks) so the export is actually complete and portable, matching GDPR Art. 20 'all data concerning the data subject'.
- Standard-format export for activity/GPS data (Endurain/FitTrackee): in addition to the proprietary JSON, emit GPS activities in an interoperable format (GPX/TCX/FIT) so users can take their training history to another tool — true data portability, not just a backup of our internal shape.
- Erasure must clean up out-of-DB artifacts too: deletion should remove on-disk uploaded files (avatars under uploads/avatars, post photos under uploads/posts, any GPS files), revoke external integrations (Strava OAuth token revocation, RevenueCat/billing), and best-effort delete any export archives previously generated for the user. DB-only deletion leaves personal data behind.
- Append-only consent history rather than last-state-only: a GDPR Art. 7 'verifiable consent' ledger should retain every grant/revoke event with timestamp + version, so you can later prove WHEN a user consented to a given data type and WHEN they withdrew it. An upsert that overwrites the prior decision loses that audit trail.
- Two-step / re-authenticated destructive actions: account deletion (and ideally export of sensitive data) should require a fresh credential check or typed confirmation, mirroring how change-password already requires the current password — protects against a stolen/active session nuking the account.
- Soft-delete grace window with scheduled hard-erase: reserve a short reversible window (account marked deleted + access revoked immediately, hard cascade run by a job after N days) so an accidental or coerced deletion can be undone, while still honoring the GDPR ~30-day erasure SLA. The schema already has User.status with a 'deleted' value that is currently unused.
- Deletion receipt / audit record: persist a minimal, non-PII tombstone (anonymized userId hash + timestamp + 'erased') outside the cascade so you can prove the erasure happened for compliance, without retaining the person's data.
- Per-activity / per-post visibility override at creation and edit time (both references support this): allow each post/GPS activity to carry its own visibility independent of the profile default, and surface it clearly in the UI — ZVELT has post.visibility but should ensure it is settable per-item and respected uniformly across feed, profile, direct link, and search.
- Export completeness contract enforced by test: a test that diffs the set of user-owned Prisma relations against the export's select list (and the erase cascade's delete list) so that adding a new user-linked table fails CI until it is added to BOTH export and erasure — prevents silent gaps as the schema grows.

**Where ZVELT is already as-good/better:**
- Atomic single-transaction erasure with explicit children-before-parents ordering and a 500 ERASURE_FAILED rollback path — stronger guarantee than a naive multi-step delete; well unit-tested for ordering and atomicity (gdpr.test.ts).
- Credential-leak-proof export with a dedicated test asserting authIdentities/refreshTokens/passwordHash are never in the SELECT projection nor in the serialized body — explicit negative-space testing many projects lack.
- 404-not-403 privacy gate so forbidden posts are byte-identical to missing ones (no existence enumeration), with a test proving the two responses are indistinguishable — a privacy nuance most fitness apps miss.
- Per-data-type health consent with granted/revoked timestamps and consentVersion already modeled and routed — finer-grained than a single blanket health toggle.
- Privacy-by-default baked into schema defaults (discoveryOptIn=false, feedFriendsOnly=true, privacyDefault='friends', dmFriendsOnly=true) rather than opt-out — aligns with CLAUDE.md principle 1 and is stricter than typical defaults.
- Synchronous, immediate self-service export (no waiting on an async job/email) is a better UX for small accounts than FitTrackee's queued-only flow — worth keeping as the fast path even after adding async for large accounts.

**Missing edge cases / test gaps:**
- Erasure does not delete on-disk files: avatars (uploads/avatars/<userId>.ext) and post photos (uploads/posts/<postId>.ext) survive deletion — personal images remain after right-to-erasure. No test covers file cleanup.
- Erasure does not revoke external integrations: Strava OAuth tokens, RevenueCat/billing linkage, and push tokens at the FCM provider are not revoked/cleaned remotely — only local rows removed. Personal data persists at third parties.
- Export omits health-consent records and may omit some user-linked tables: the profile export select list does not include healthConsents, and there is no test guaranteeing the export covers every user relation — risk of silently incomplete exports as schema grows (challenge*, wearableConnections, healthImports, healthDailyMetrics are in export but no completeness assertion ties select↔schema).
- Export is unbounded/synchronous: the whole object graph is loaded in one findUnique with no pagination/streaming — a power user with thousands of sets/GPS points can blow the request timeout or memory. No test for large-account behavior.
- Export contains no media bytes or links to media — uploaded photos and GPS tracks are not in the export, so the 'portable copy' is incomplete vs GDPR Art. 20.
- No re-authentication or typed-confirmation on DELETE /v1/me/account — an active stolen session can irreversibly erase the account.
- Erasure is immediate and irreversible — no grace/undo window; User.status='deleted' exists in schema but is never used; accidental deletion is unrecoverable.
- Consent ledger keeps only the latest decision per type (upsert overwrites grantedAt/revokedAt) — cannot prove the full grant→revoke→regrant history required for verifiable consent.
- No deletion audit/tombstone — after erasure there is no record that the request was fulfilled (hard to prove compliance to a regulator).
- Partial-failure on export not handled distinctly from success: if a single relation read fails the whole 200 fails, but there is no per-section error reporting; conversely a relation silently missing from select produces a 'successful' but incomplete export with no warning.
- Direct conversations are deleted wholesale on erasure (directMessage + directConversation by either participant) — this also destroys the OTHER participant's copy of the conversation thread; need a policy decision (anonymize the leaving user's messages vs delete the whole thread) to avoid erasing a non-consenting third party's data.

### progressive-overload-programming
**References:** Liftosaur (Liftoscript progression engine, plate calculator, warmup syntax, rest timers; AGPL) — studied via public docs only; wger (routine progression rules, prescribed-vs-logged model, RestConfig, supersets/alternatives; AGPL) — studied via public readthedocs only. No source code, schema, or assets were read or ported.

**ZVELT now:** ZVELT has a deterministic, RPE-gated autoregulation engine the references mostly lack. backend/src/lib/progressive-overload.ts: autoregulateLoad() reads the last set's max RPE and chooses deload (RPE>=9.5, -10%), hold (RPE>8.0), normal bump, or accelerate (RPE<=6.0, 1.5x bump); bumpForLevel() gives level-aware increments (beginner +2.5kg compound / 2.5% accessory, intermediate +1.25kg / 1%, advanced 0 = needs periodization); roundToPlate() snaps to a 2.5kg step with a 5kg floor; computeProgressiveLoads() batches per-exercise decisions (BW_REPS lifts get +1 rep not load; 30-day stall detection via UserExerciseRank holds weight). backend/src/lib/progression-context.ts: getRecentProgression() + formatProgressionForPrompt() brief the AI with top-12 actively-trained lifts, best e1RM, last weight x reps, days-since-last, and 30-day e1RM delta. backend/src/services/weekly-plan.service.ts: generateAndPersistWeeklyPlan() unifies plan creation, attaches suggestedWeightKg/loadSource/loadReason per exercise to PlannedWorkout.exercisesJson. backend/src/services/session-reconcile.service.ts: recordWorkoutProgress() writes durable per-lift state to UserExerciseProgress on workout complete (last weight/reps, hardest RPE, next source/weight/reason). Schema: PlannedWorkout (exercisesJson stores sets/reps/restSeconds/notes/whyThisExercise), WorkoutSet (tag WORK|WARMUP|DROP, rpe, isCompleted), UserExerciseProgress (durable autoreg state). App: rest_interval_store.dart logs actual rest BETWEEN sets for analytics; program_builder_screen.dart is an AI-prompt wizard (goal/level/days/equipment -> weekly plan), not a deterministic rule editor. Pure helpers are unit-tested in progressive-overload.test.ts.


**Strong ideas (clean-room):**
- Configurable, named progression SCHEMES selectable per exercise/program instead of one global RPE rule. ZVELT today hard-codes a single autoregulation policy. Liftosaur exposes three schemes: linear progression (add fixed load after N successes / back off after N failures), double progression (climb reps within a band e.g. 8->12, then reset reps and add load), and reps-sum (add load when total reps across all working sets crosses a threshold). Requirement: let a program/exercise carry a progressionScheme enum + params, and have the engine dispatch to the matching policy, with RPE-autoregulation remaining the default/overlay.
- Multi-session state machine for double-progression. ZVELT's autoreg is single-session (reads only the last set). Double progression requires remembering where the rep climb is across sessions. ZVELT already has the durable UserExerciseProgress table written on complete but it is NEVER READ back. Requirement: feed UserExerciseProgress (last reps/weight + a rep-band cursor) into computeProgressiveLoads so a lift currently mid-band gets a +1 rep prescription, and only converts to a load bump + rep reset once the top of the band is hit at acceptable RPE.
- Explicit DELOAD WEEKS (block-level), distinct from ZVELT's per-set RPE deload. Liftosaur and wger both support disabling progression for a whole day/week (progress:none) and periodized cycles. Requirement: tag a week (or a recurring cadence e.g. every 4th week) as a deload where the planner prescribes reduced load/volume and suppresses bumps, especially for advanced lifters where bumpForLevel returns 0 today and they get no structured progression at all.
- Prescribed-vs-actual reconciliation as a first-class concept (from wger). wger stores both the prescribed config and the logged actual, and progression CONDITIONS compare them ('only add load if the prescribed reps were actually hit'). ZVELT stores prescribed values in exercisesJson and actuals in WorkoutSet but the progression decision only looks at actuals. Requirement: gate progression on whether the user MET the prescription (hit target reps at/under target RPE), holding when they fell short rather than blindly bumping or holding off raw RPE.
- Warmup-set GENERATION from the working weight. Neither ZVELT plan nor tracker generates warmup sets today (WorkoutSet has a WARMUP tag but nothing produces them). Liftosaur derives a warmup ramp as percentages of the first working set. Requirement: given a target working load, generate a short ascending warmup ramp (e.g. empty bar, ~40/60/80% for 5/3/2 reps) only for barbell/compound lifts, plate-rounded, surfaced as WARMUP-tagged sets in the tracker and excluded from e1RM/ranking (which ZVELT already excludes by tag).
- Plate calculator surfaced at the point of lifting. ZVELT rounds loads to a 2.5kg step but never tells the user which plates to load, and has no concept of available plates/bar weight. Liftosaur shows the per-side plate breakdown and rounds to plates the user actually owns. Requirement: a pure plate-math function (bar weight + available plate inventory -> per-side stack + achievable weight) plus a per-user equipment/plate-inventory setting, used both to display the stack in the tracker and to constrain roundToPlate to achievable loads.
- Exercise substitution with progression carry-over. ZVELT resolves AI exercise names but offers no swap UX, and wger models exercise alternatives. Requirement: let a user swap a prescribed exercise for a same-pattern alternative (same movementPattern, similar rankModel) and carry the progression intent across (start the substitute at a sensible fraction of the original's e1RM rather than from no_history), so a swap doesn't reset the user to 'pick on feel'.
- Per-set rest-time prescription with an in-app countdown driven by plan restSeconds. ZVELT already PRESCRIBES restSeconds in exercisesJson and TRACKS actual rest via rest_interval_store, but the two are disconnected — there is no countdown timer that fires from the prescribed value. Requirement: surface a rest countdown seeded by the prescription, compare actual vs prescribed (wger's RestConfig pattern), and feed adherence back into explainability ('you rested 45s vs 90s prescribed — that can inflate RPE').

**Where ZVELT is already as-good/better:**
- RPE-gated autoregulation is more sophisticated than Liftosaur's default schemes and wger's static rules: ZVELT modulates the jump (deload/hold/normal/accelerate) off the actual hardest RPE of the session rather than only success/fail counts. Liftosaur requires the user to hand-script RPE logic in Liftoscript; ZVELT does it automatically.
- Explainability is built in (Zvelt principle #3): every load decision carries a human-readable reason string (loadReason) surfaced to the UI. Liftosaur/wger expose the rule but not a natural-language 'why this weight today'.
- Level-aware progression curve out of the box (bumpForLevel) — beginner/intermediate/advanced get different increments automatically, and advanced correctly stops linear progression. References make the user encode this manually.
- Safe-by-default load picking: AI chooses exercises but a deterministic engine chooses LOAD, with plate rounding, a 5kg floor, and no_history fallbacks — avoids hallucinated unsafe prescriptions. This AI+deterministic split is an architecture neither reference has.
- Bodyweight lifts handled distinctly (BW_REPS -> rep progression, capped at 15) rather than forcing a weight model onto them.
- Durable autoreg state already persisted (UserExerciseProgress) and idempotent on workout complete — the storage substrate for multi-session progression already exists; references store state but ZVELT's is already integrated with the ranking/e1RM pipeline.
- Pure, unit-tested decision helpers (progressive-overload.test.ts covers bumpForLevel, roundToPlate, autoregulateLoad edge cases incl. deload plate rounding) — a test discipline that de-risks future scheme additions.

**Missing edge cases / test gaps:**
- UserExerciseProgress is WRITTEN on complete but NEVER READ — next-week planning still re-queries last sets via computeProgressiveLoads, so the durable nextWeightKg/nextSource the engine computed is silently discarded. No test asserts the table is consumed.
- No deload-WEEK concept: advanced lifters (bumpForLevel returns 0) get zero structured progression — they hold forever with no periodized over-reach/deload cycle.
- Double/rep-band progression impossible: engine reads only the last set, so it cannot 'climb reps then add weight'. A lift stuck at the top of a rep range never converts reps->load.

- No warmup generation despite a WARMUP set tag existing — users get a working weight with no ramp, and the tag is effectively unused by the planner.
- No plate calculator / available-plate inventory — roundToPlate can suggest a 2.5kg-stepped weight the user physically cannot load (e.g. only has 5kg plates), and never shows the per-side stack.
- Prescribed restSeconds is stored but no countdown consumes it; actual rest is tracked separately with no adherence comparison, so RPE inflation from short rests is invisible to the engine.
- No exercise substitution: swapping a lift drops the user to no_history ('pick on feel'), losing progression continuity for an equivalent movement.
- ZVELT_AUTOREG=off kill-switch path (forces always-bump) has no dedicated test asserting it reproduces legacy behavior.
- Deload only ever goes -10% once; there is no escalating back-off if the user keeps failing across multiple sessions (Liftosaur supports N-failures-then-reduce).

**Strong ideas (clean-room):**
- Server-side file import of FIT/GPX/TCX as a first-class path. Endurain treats uploaded activity files (FIT preferred) as a primary ingestion source via dedicated parsers. ZVELT has NO server-side file parsing — it can only ingest via on-device Health APIs or the Terra webhook. Requirement: accept a user-uploaded .fit/.gpx/.tcx file, parse it into ZVELT's normalized activity shape (samples + summary), and store it as a UserHealthImport/activity with provenance 'file_upload'. This unlocks users migrating off Garmin/Strava/Coros with no live integration.
- Bulk / multi-file import as an explicit, resumable operation. Endurain documents bulk upload of many files. Requirement: a batch import that accepts N files, processes them as independent units, reports per-file success/failure, and is idempotent so re-uploading the same archive does not create duplicates.
- Scheduled polling as a fallback to webhook-only sync. Endurain polls Garmin hourly (activities) / four-hourly (body composition) and Strava hourly, on startup and on interval. ZVELT's aggregator path is webhook-only — if a Terra webhook is dropped, that data is lost forever with no catch-up. Requirement: a periodic reconciliation/backfill job per connected provider that pulls a recent window and relies on the existing upsert key to no-op already-seen items, so a missed webhook self-heals.
- Idempotent, retry-safe webhook ingestion with an inbox/outbox pattern. Endurain's polling model is inherently retryable; ZVELT's webhook handler persists inline and swallows DB failures with `.catch(() => {})`, so a transient DB error silently drops the payload and Terra will not redeliver. Requirement: persist the raw webhook envelope immediately (cheap, append-only), ACK fast, then process asynchronously with bounded retries and a dead-letter record; never lose a payload on a transient failure.
- Cross-source deduplication of the SAME activity arriving from multiple paths. A single run can land via on-device Health Connect AND the Terra aggregator AND (future) a file upload, producing three rows for one workout. Endurain's whole premise is one canonical activity per real-world event regardless of source. Requirement: a dedup key based on activity identity (athlete + start-time window + activity type + duration/distance tolerance), not just per-source external id, so overlapping records from different providers collapse to one canonical activity with the others kept as linked alternates.
- Activity normalization into a canonical model (summary + sample stream). Endurain normalizes GPX/TCX/FIT into one internal activity representation regardless of source library. ZVELT stores provider-shaped JSON blobs in `user_health_imports.payload` with no normalized summary fields for distance/duration/elevation/avg-HR on an activity. Requirement: define a canonical activity summary (type, start/end, distance, duration, elevation gain, avg/max HR, calories, source provenance) plus an optional sample/track stream, populated by every ingestion path.
- Derive the daily rollup that the schema already promises. `UserHealthDailyMetric` exists and its own comment says daily summaries 'should be derived from this table' — but no code populates it. Endurain derives clean per-day metrics from raw imports. Requirement: a reconciliation job that folds `user_health_imports` into `user_health_daily_metrics` per (day, sourcePath, provider) and then into one user-facing daily figure, picking a winning source per metric to avoid double-counting overlapping providers.
- Unlink = data deletion contract per provider terms. Endurain explicitly deletes all Strava-imported activities/gear on unlink, per Strava's API Agreement. ZVELT's DELETE /:provider is a 204 no-op that leaves imported rows behind. Requirement: disconnecting a provider purges (or anonymizes) data attributed to that provider's sourcePath, satisfying both Strava/Garmin platform agreements and ZVELT's own GDPR erasure stance.
- Surface partial-success/failure detail to the user. Endurain's per-file import and the Garmin 'missing Speed data' issue show real-world files fail individually. Requirement: an import-status record per upload/sync run (queued/processing/done/failed + reason) the app can poll, instead of ZVELT's current opaque webhook flow that returns only an aggregate `imported` count.

**Where ZVELT is already as-good/better:**
- On-device universal ingestion via HealthKit/Health Connect covers a far broader, OEM-agnostic device surface than Endurain's two named integrations (Strava + Garmin). ZVELT auto-picks up Samsung, Xiaomi/Mi, Fitbit, Google, etc. through Health Connect with zero per-vendor credentials.
- Robust partial-permission handling. `_readSafe` + the core-types check (`_coreTypes`) correctly survive Health Connect's partial-grant model and the documented 'connected then disconnected next launch' class of bug. Endurain's credential-based polling has no equivalent fine-grained consent surface.
- Sleep overlap-merge dedup (`_unionDurationHours`) is a concrete, tested-in-practice fix for multi-source double-counting (the '27h night' bug) — a more sophisticated same-event reconciliation than a simple per-source key.
- GDPR-grade per-type consent ledger (`HealthConsent`) with grant/revoke timestamps and source — privacy controls beyond Endurain's 'self-hosted = you own it' posture.
- Encrypted on-device cache (SQLCipher) for the most sensitive data, plus the resilient backfill-not-marked-done-on-empty pattern that self-heals when an upstream source (e.g. Samsung Health) hasn't synced yet.
- Webhook signature verification is timing-safe HMAC over `t.rawBody` with length-checked compare — a real security control Endurain's polling model doesn't need but which ZVELT already does correctly.
- GPS noise filtering in `route_tracker.dart` (accuracy gate, displacement floor, teleport guard, elevation hysteresis) is a mature live-capture pipeline that file-import-centric Endurain does not have.

**Missing edge cases / test gaps:**
- Dropped/duplicate webhook delivery: Terra may redeliver or miss; ZVELT has no replay log and silently drops on transient DB error (`.catch(() => {})`). No scheduled catch-up means a missed webhook is permanent data loss.
- Same activity from multiple sources: an indoor run logged by watch reaches ZVELT via Health Connect AND Terra; current keys (per-source externalId) create two rows with no cross-source collapse.
- On-device imports never reach the backend: `health_imported_records` lives only in local sqflite; there is no upload path, so the backend's `user_health_imports`/daily metrics are blind to Health Connect/HealthKit data entirely.
- Empty/`UserHealthDailyMetric` is never written: the rollup table and its unique key exist but nothing populates it, so any server-side daily analytics over wearable data are impossible today.
- Garmin Training Center / malformed files: Endurain's issue #299 shows real files missing Speed/required fields; ZVELT has no file parser at all, so the moment file import is added this must be handled per-field, per-file, not all-or-nothing.
- Timezone of activity vs civil day: `activities.ts` calendar takes a tzOffset, but imported health data start/end are stored UTC with no per-record offset, so day-bucketing of imported activities can land on the wrong calendar day.
- Provider unlink leaves orphaned data: DELETE is a no-op; imported rows and the connection's derived metrics persist, violating Strava/Garmin agreements and the app's erasure promise.
- Unit/canonicalization drift on import: incoming distances/weights arrive in provider units; without a normalization step into kg/meters the rollup and ranking math can mis-scale.
- Idempotency on bulk re-upload: re-uploading the same FIT archive (common user behavior) must not duplicate; needs a content/identity hash, which doesn't exist yet.
- Backfill window vs retention: a brand-new aggregator connection should pull historical data, but the webhook-only path explicitly returns imported:0 and relies entirely on Terra pushing history — untested if that push is partial.

### workout-logging-offline-sync
**References:** wger (wger-project/flutter + REST backend) and Liftosaur (offline execution/history, public "Offline mode in Liftosaur" blog + docs). Studied only public README/docs/blog/observable behavior — no source read or ported.

**ZVELT now:** ZVELT has a focused, well-tested offline-set path. `app/lib/services/offline_set_queue.dart` persists pending SET ADDS in SharedPreferences keyed per-user (`zvelt_offline_set_queue_v1_<userId>`), with a serialized read-modify-write lock (`_synchronized`) so concurrent enqueues don't clobber, exponential backoff (2^n s capped 30 min) via `_afterTransientFailure`, and a flush that classifies outcomes: 4xx => drop + Crashlytics report, 5xx/network => keep with backoff, in-window => deferred (FlushResult synced/dropped/deferred). Legacy entries get a DETERMINISTIC content-derived clientSetId so a re-flush can't mint a fresh UUID and duplicate. `offline_sync_coordinator.dart` is a singleton bridging connectivity_plus to the queue: auto-flush on offline->online transition, single-flight `_flushing` guard, `pendingCount` ValueNotifier for UI. The tracker (`workout_tracker_screen.dart`) generates one client UUID per set, does optimistic in-place patching (`_patchWorkoutSets`), and on any addSet/updateSet failure enqueues offline. Backend `backend/src/routes/workouts.ts` AddSetSchema accepts optional `clientSetId` (uuid) and dedupes: pre-check findFirst, plus P2002 race recovery returning the existing row with `idempotent:true`. Schema (`schema.prisma`) WorkoutSet.clientSetId is `@unique` (GLOBAL, not per-user). PATCH set has anti-cheat >2x note gate + before/after SetEditAudit. List endpoint paginates (page/limit, max 50). `workout_draft_store.dart` stores only a lightweight resume POINTER (workoutId/title/savedAt/counts, 48h stale TTL) — not full offline set data. Tests in `app/test/offline_set_queue_test.dart` cover roundtrip, legacy determinism, 4xx/5xx/deferred, concurrent enqueue.


**Strong ideas (clean-room):**
- Full local-first session store, not just a queue of adds: wger and Liftosaur keep the entire in-progress (and recent) workout durably on-device so the screen renders and the user can ADD, EDIT, REORDER and DELETE sets/exercises fully offline, then reconcile on reconnect. ZVELT's draft store is only a pointer, so a tracker opened with no network shows a spinner/error and edits-while-offline are not all queued. Requirement: persist the full workout structure (exercises + sets + local edits) locally as the source of truth for the active session, with the server as a sync target.
- Cover the WHOLE mutation set offline, not only set-add. References sync edits and deletes too. ZVELT only enqueues addSet on failure; updateSet/deleteSet/addExercise/complete failures are NOT queued (updateSet failure currently re-routes to an ADD enqueue, which can create a duplicate set rather than apply the edit). Requirement: an operation log that records the operation TYPE (add/update/delete/complete) per entity with idempotency keys, replayed in order.
- Idempotency keyed by stable client-generated entity IDs for ALL entities, not just sets. References let the client mint IDs so create+update+delete of the same item reconcile deterministically. ZVELT only has clientSetId; a set EDIT made offline then replayed has no idempotency story. Requirement: client-side stable ids for workout/exercise/set and op-level dedup tokens so replays are exactly-once.
- Explicit ordered replay with dependency awareness: a set add depends on its workoutExercise existing server-side; an edit depends on the add having synced. Liftosaur/wger reconcile a coherent state, not independent rows. ZVELT flushes sets independently and would 404 if the parent exercise was itself only created offline. Requirement: topologically order the replay (create parents before children) or attach the operations so the server can create the graph atomically.
- Conflict resolution policy made explicit and documented: Liftosaur uses date-keyed history concatenation + 'local is newer' last-write-wins and openly accepts rare cross-device loss; wger does optimistic update + conflict resolution on same-entry multi-device edits. ZVELT has no defined policy for 'set edited on phone offline + edited on watch/web' — last successful PATCH silently wins. Requirement: choose and document a deterministic policy (e.g. last-write-wins by client timestamp, or server-authoritative with a surfaced 'changed elsewhere' notice).
- Schema-versioned local payload with migration-on-load: Liftosaur version-stamps persisted state and migrates both sides before merging so an app update never corrupts queued data. ZVELT keys are *_v1 but there is no migration step — a shape change would silently drop entries via the try/catch return []. Requirement: version the persisted queue/draft payload and migrate forward on load instead of discarding.
- Surface sync state and failures to the user, not just Crashlytics: references show pending/synced indicators. ZVELT drops 4xx sets to Crashlytics only — the user never learns a logged set vanished. Requirement: a visible 'N sets pending / failed to sync' affordance with a tap-to-review/retry, so silent data loss becomes user-recoverable.
- Background sync independent of an open screen: wger auto-syncs via background services so logs flush even if the app is backgrounded. ZVELT's coordinator only flushes on a connectivity event while the process is alive and `start()` was called. Requirement: a periodic/background flush trigger (app-resume + interval) so a queue doesn't sit until the next connectivity flip.

**Where ZVELT is already as-good/better:**
- Server-side idempotency is stronger than Liftosaur's: ZVELT has a true DB unique constraint on clientSetId plus P2002 race recovery returning the existing row — Liftosaur relies on date-keyed array dedup that explicitly 'sacrifices correctness for simplicity'.
- Principled transient-vs-permanent error classification: 4xx dropped (with audit), 5xx/network kept with capped exponential backoff and a deferred state — Liftosaur's blog describes no retry/backoff queue at all (real-time sync on state change).
- Per-user namespaced offline storage keys with the key resolved ONCE per mutation to survive a mid-op user/token flip — more careful than the single-user assumptions typical of these reference apps.
- Serialized read-modify-write lock preventing concurrent-enqueue clobber, pinned by a dedicated test — a concrete correctness guarantee the references' 'local is newer overwrite' model does not make.
- Deterministic legacy-id derivation so pre-idempotency queued entries can't duplicate on replay — a thoughtful backward-compat detail.
- Anti-cheat audit trail on set edits (before/after SetEditAudit + >2x note gate) that fitness trackers like Liftosaur/wger do not have — ZVELT's edit path is more accountable.

**Missing edge cases / test gaps:**
- updateSet failure falls back to enqueueing an ADD (a new clientSetId) instead of replaying the EDIT — on reconnect this can create a duplicate set rather than update the intended one.
- deleteSet and addExercise have no offline queue at all — failures while offline are surfaced as errors and lost.
- Parent-not-yet-synced: a set queued against a workoutExercise that was itself only created offline will 404 on flush and be permanently DROPPED as a 4xx.
- Opening the tracker with no connectivity: _load() does a GET and shows an error/spinner — the in-progress session is not renderable offline because the draft store holds only a pointer.
- clientSetId is GLOBALLY @unique, not scoped per user/workout — a (astronomically unlikely) UUID collision across users would map a flushed set into a findFirst that is user-scoped, causing a P2002 throw with no existing row found for THIS user, surfacing as a 500.
- No cap or TTL on the offline queue: a long outage with many sets grows SharedPreferences unbounded; entries never expire (unlike the 48h draft TTL).
- Persisted-payload shape change has no migration — corrupt/old JSON is silently swallowed to [] (loss), not migrated.
- complete-workout offline is not queued; completing while offline just errors and the session stays a draft with no retry.
- No surfaced pending/failed UI on the tracker itself — pendingCount notifier exists but the tracker screen does not show it, so the user can't tell sets are unsynced.
- Cross-device concurrent edit (phone offline + watch online) has no conflict policy or test — last PATCH silently wins.
- No test that a 5xx then a later successful flush actually syncs the SAME clientSetId (end-to-end backoff->recover), and no test for parent-404 drop behavior.
